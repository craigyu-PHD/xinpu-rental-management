# 套房出租管理系統｜網頁 App 功能需求規格

## 一、文件目的

本文件用於整理「套房出租管理系統」之網頁 App 功能需求，供後續代理人 AI、工程師或低程式碼工具依照本文件開發實用版本。

本系統定位為「小型房東自用雲端管理系統」，主要用於管理 28 號 3 樓之 4 間套房，包含 301、302、303、304 四間房之房間、租客、租約、收租、電費、支出、維修、附件與報表。

本系統第一版以房東自用為主，不優先開放房客登入、線上付款、電子簽約與自動通知等複雜功能。

---

## 二、整體管理原則

### 2.1 管理主軸

系統應採用以下管理邏輯：

```text
以房間為單位
以租約為依據
以月份為帳務週期
以收入、支出、代收款、押金四類分帳
以維修紀錄串接照片與支出
以雲端附件保存證據
最後自動產生月報與年報
```

### 2.2 房間範圍

系統預設管理 4 間套房：

| 房號 | 說明 |
|---|---|
| 301 | 套房一 |
| 302 | 套房二 |
| 303 | 套房三 |
| 304 | 套房四 |

### 2.3 費用負擔規則

| 項目 | 負擔者 | 系統定位 |
|---|---|---|
| 房租 | 房客支付 | 正式租金收入 |
| 電費 | 各房客依臺電帳單與各房電表比例負擔 | 電費代收／應收款 |
| 水費 | 房東支應，已含在房租內 | 房東營運支出 |
| 垃圾清潔費 | 房東支應，已含在房租內 | 房東營運支出 |
| 維修費 | 依責任歸屬判斷 | 房東支出、房客負擔或押金扣抵 |
| 押金 | 房客交付、房東保管 | 暫收款，不列為租金收入 |

### 2.4 帳務分類原則

系統內所有金流應至少區分為四大類：

| 分類 | 說明 |
|---|---|
| 租金收入 | 房客每月繳納之房租，計入出租收入 |
| 電費代收 | 房客依臺電帳單分攤之電費，原則上不列入租金收入 |
| 支出紀錄 | 水費、垃圾清潔費、維修、設備、耗材、管理相關支出 |
| 押金紀錄 | 押金收取、扣抵與退還，不直接列入一般收入 |

# 四、各分頁功能需求

## 4.1 儀表板

### 功能文字說明

儀表板是系統首頁，用於快速查看目前出租狀況、收租狀態、電費收款狀態、本月支出、本月淨利與待處理維修。使用者打開系統後，應能一眼判斷本月哪些房間已繳房租、哪些尚未繳清、哪些電費未收、是否有維修案件尚未處理。

### 功能細項設定

1. 顯示本月應收房租總額。
2. 顯示本月已收房租總額。
3. 顯示本月未收房租總額。
4. 顯示本月電費應收總額。
5. 顯示本月電費已收總額。
6. 顯示本月電費未收總額。
7. 顯示本月支出總額。
8. 顯示本月出租淨利。
9. 顯示待處理維修案件數量。
10. 顯示即將到期租約。
11. 顯示各房狀態，例如出租中、空房、整修中、即將退租。
12. 顯示提醒卡片，例如「302 本月房租未收」、「304 電費部分收款」、「301 維修待處理」。

### 需求說明

1. 儀表板應以卡片式呈現，避免文字過多。
2. 儀表板資料需由各分頁資料自動彙整，不應手動輸入。
3. 本月淨利計算時，應以房租收入扣除房東負擔支出，不應將電費代收直接視為租金收入。
4. 儀表板需支援月份切換，方便查看過去月份資料。
5. 儀表板應可點擊卡片進入相對應分頁，例如點擊「未收房租」可進入收租管理。

## 4.2 房間管理

### 功能文字說明

房間管理用於建立與維護 301、302、303、304 四間套房資料。房間是本系統的核心單位，所有租客、租約、房租、電費、維修、支出與附件都應能對應到特定房號。

### 功能細項設定

每一間房應至少包含下列欄位：

| 欄位 | 說明 |
|---|---|
| 房號 | 301、302、303、304 |
| 房間名稱 | 可設定顯示名稱，例如「301 套房」 |
| 房間狀態 | 出租中、空房、整修中、即將退租 |
| 預設月租金 | 該房標準租金 |
| 預設押金 | 該房標準押金 |
| 收租日 | 每月幾號收租 |
| 電費規則 | 臺電帳單分攤模式 |
| 水費規則 | 房東支應，含在房租內 |
| 垃圾清潔費規則 | 房東支應，含在房租內 |
| 房間設備 | 冷氣、熱水器、床、衣櫃、冰箱、桌椅等 |
| 備註 | 房間特殊情況 |

### 需求說明

1. 第一版系統可預設 4 間房，不必設計成大量物件管理系統。
2. 每間房應可查看目前租客、目前租約、近期收租狀態、近期電費與維修紀錄。
3. 若房間狀態為空房，仍可登記房東負擔之支出，例如整修費、清潔費、空房期間用電。
4. 房間資料不可因租客退租而刪除，應保留歷史紀錄。
5. 房間應作為系統各資料表的主要關聯欄位。

## 4.3 租客管理

### 功能文字說明

租客管理用於記錄每位租客的基本資料、聯絡方式、入住紀錄與備註。租客資料應獨立於房間資料，避免換租客時覆蓋舊資料。

### 功能細項設定

每位租客應至少包含下列欄位：

| 欄位 | 說明 |
|---|---|
| 租客姓名 | 主承租人姓名 |
| 聯絡電話 | 手機號碼 |
| LINE 或其他聯絡方式 | 可選填 |
| 職業或任職單位 | 可選填 |
| 緊急聯絡人 | 姓名與電話 |
| 身分證資訊 | 可選填，需注意個資保護 |
| 目前承租房號 | 自租約帶入 |
| 入住日期 | 自租約帶入或手動輸入 |
| 退租日期 | 退租後記錄 |
| 備註 | 特殊約定、聯絡紀錄、租客習慣等 |
| 附件 | 證件照片、入住點交照片等 |

### 需求說明

1. 租客資料應可保留歷史，退租後不應刪除。
2. 同一租客未來若再次入住，應可建立新租約而非重建租客資料。
3. 租客頁面應可查看其歷史租約、繳租紀錄、電費紀錄、維修紀錄與押金紀錄。
4. 涉及個資的欄位應盡量簡化，避免過度蒐集。
5. 附件應與租客資料綁定，但也應可在附件中心搜尋。

## 4.4 租約管理

### 功能文字說明

租約管理用於管理租客與房間之間的承租關係。租約是房租、押金、收租日與合約期限的依據。房間不等於租約，租客也不等於租約；每次入住都應建立一份租約紀錄。

### 功能細項設定

每份租約應至少包含下列欄位：

| 欄位 | 說明 |
|---|---|
| 租約編號 | 系統自動產生 |
| 房號 | 301、302、303、304 |
| 租客 | 對應租客資料 |
| 起租日 | 合約開始日期 |
| 到期日 | 合約結束日期 |
| 月租金 | 合約約定租金 |
| 押金 | 合約約定押金 |
| 收租日 | 每月幾號收租 |
| 電費約定 | 依臺電帳單分攤，由房客負擔 |
| 水費約定 | 房東支應，含在房租內 |
| 垃圾清潔費約定 | 房東支應，含在房租內 |
| 租約狀態 | 生效中、即將到期、已到期、已退租 |
| 合約附件 | PDF、照片或掃描檔 |
| 備註 | 特殊約定 |

### 需求說明

1. 系統應依租約自動產生每月房租應收資料。
2. 租約到期前應在儀表板顯示提醒。
3. 租客退租時，租約狀態應改為已退租，但不得刪除。
4. 租約應可連結押金紀錄、收租紀錄、電費紀錄與退租紀錄。
5. 若租客中途退租，應記錄退租日期，並可進行押金結算。

## 4.5 收租管理

### 功能文字說明

收租管理用於管理每月房租之應收、實收、未收、部分收款與逾期狀態。系統應依照租約自動產生每月應收房租，再由房東登記實際收款。

### 功能細項設定

收租管理應分為「應收房租」與「實收紀錄」。

#### A. 應收房租欄位

| 欄位 | 說明 |
|---|---|
| 應收編號 | 系統自動產生 |
| 歸屬月份 | 例如 2026-04 |
| 房號 | 301、302、303、304 |
| 租客 | 對應租客 |
| 應收日期 | 依租約收租日產生 |
| 應收金額 | 月租金或按比例計算金額 |
| 收款狀態 | 未收、已收、部分收、逾期、免收 |
| 備註 | 首月按比例、補收、折讓等 |

#### B. 實收紀錄欄位

| 欄位 | 說明 |
|---|---|
| 收款日期 | 實際收到款項日期 |
| 歸屬月份 | 該筆款項屬於哪個月份 |
| 房號 | 301、302、303、304 |
| 租客 | 對應租客 |
| 收入類型 | 房租 |
| 實收金額 | 實際收到金額 |
| 付款方式 | 轉帳、現金、其他 |
| 收款附件 | 匯款截圖或收據 |
| 備註 | 補繳、提前繳、部分繳等 |

### 需求說明

1. 收款日期與歸屬月份必須分開，因為房客可能提前繳或延後繳。
2. 系統應支援部分收款，並自動計算尚未收款金額。
3. 房租收入應計入出租收入與損益報表。
4. 收款紀錄應可上傳匯款截圖或收據。
5. 已完成之收款紀錄應保留修改紀錄，避免日後帳務爭議。

## 4.6 臺電帳單管理

### 功能文字說明

臺電帳單管理用於輸入每一期臺電帳單資料。由於本出租物件與房客約定依臺電帳單計費，因此每一期電費應先建立一筆臺電帳單，再依各房電表用電比例分攤。

本系統的電費模式不是固定每度單價，而是「臺電帳單總額回推單價，再依各房用電比例分攤」。

### 功能細項設定

每一期臺電帳單應至少包含下列欄位：

| 欄位 | 說明 |
|---|---|
| 電費帳單編號 | 系統自動產生 |
| 帳單名稱 | 例如 2026 年 5–6 月臺電帳單 |
| 帳單期間起日 | 臺電帳單起始日 |
| 帳單期間迄日 | 臺電帳單結束日 |
| 公用電費 | 臺電帳單上列示之公用電費 |
| 戶內電費 | 28 號 3 樓這一戶的電費 |
| 帳單總額 | 公用電費 + 戶內電費，系統自動計算 |
| 繳費期限 | 臺電帳單繳費期限 |
| 是否已繳臺電 | 是／否 |
| 繳費日期 | 房東實際繳納臺電日期 |
| 帳單照片 | 上傳臺電帳單照片或 PDF |
| 備註 | 補充說明 |

### 需求說明

1. 臺電帳單總額應由系統自動計算，不應手動輸入後覆蓋。
2. 臺電帳單資料建立後，應進入電表抄表管理，輸入各房電表度數。
3. 每一期臺電帳單應可查看各房分攤結果與收款狀態。
4. 臺電帳單照片應與該期帳單綁定，方便日後查證。
5. 若帳單已完成分攤並產生應收款，系統應限制任意刪除或修改。

## 4.7 電表抄表管理

### 功能文字說明

電表抄表管理用於記錄 301、302、303、304 各房在每一期臺電帳單期間的電表度數。系統應依上期度數與本期度數自動計算各房使用度數，作為電費分攤依據。

### 功能細項設定

每一期每間房應包含下列欄位：

| 欄位 | 說明 |
|---|---|
| 電費帳單編號 | 對應臺電帳單 |
| 房號 | 301、302、303、304 |
| 上期度數 | 系統可自上期本期度數自動帶入 |
| 本期度數 | 房東本期抄表輸入 |
| 使用度數 | 本期度數 - 上期度數，系統自動計算 |
| 電表照片 | 上傳本期電表照片 |
| 負擔者 | 房客、房東、前租客、新租客 |
| 備註 | 換表、異常、空房期間用電等 |

### 需求說明

1. 系統應自動帶入上期度數，減少人工輸入錯誤。
2. 若本期度數小於上期度數，系統應警示，避免直接結算。
3. 每房本期電表照片應能上傳，作為向房客說明的依據。
4. 若遇到換表，系統應允許建立換表紀錄並手動調整使用度數。
5. 若遇到中途退租或入住，系統應允許在同一期帳單內記錄分段用電。

## 4.8 電費分攤管理

### 功能文字說明

電費分攤管理是本系統的客製化核心功能。系統應將臺電帳單總額依照各房用電度數比例進行分攤，並同時顯示各房負擔之戶內電費分攤、公用電費分攤與應繳總電費。

### 計算規則

假設：

| 代號 | 意義 |
|---|---|
| A | 臺電帳單上的公用電費 |
| B | 臺電帳單上 28 號 3 樓戶內電費 |
| T | 本期總電費，T = A + B |
| U301 | 301 房本期用電度數 |
| U302 | 302 房本期用電度數 |
| U303 | 303 房本期用電度數 |
| U304 | 304 房本期用電度數 |
| U | 四房合計用電度數 |

系統應使用下列邏輯：

```text
本期總電費 T = 公用電費 A + 戶內電費 B
全戶總用電 U = U301 + U302 + U303 + U304
各房用電比例 = 各房用電度數 ÷ 全戶總用電度數
各房戶內電費分攤 = 戶內電費 B × 各房用電比例
各房公用電費分攤 = 公用電費 A × 各房用電比例
各房應繳電費 = 各房戶內電費分攤 + 各房公用電費分攤
```

### 功能細項設定

每一期每間房應產生下列分攤資料：

| 欄位 | 說明 |
|---|---|
| 電費帳單編號 | 對應臺電帳單 |
| 房號 | 301、302、303、304 |
| 使用度數 | 由電表抄表帶入 |
| 用電比例 | 該房使用度數 ÷ 四房總使用度數 |
| 戶內電費分攤 | 戶內電費 × 用電比例 |
| 公用電費分攤 | 公用電費 × 用電比例 |
| 應繳電費 | 戶內電費分攤 + 公用電費分攤 |
| 尾差調整 | 因四捨五入產生之調整金額 |
| 最終應收電費 | 應繳電費 + 尾差調整 |
| 實收電費 | 房客實際繳納金額 |
| 收款狀態 | 未收、已收、部分收、逾期 |
| 收款日期 | 實際收款日期 |
| 收款附件 | 匯款截圖或收據 |
| 備註 | 補充說明 |

### 尾差處理規則

1. 各房應繳電費可四捨五入至整數元。
2. 四房加總後若與臺電帳單總額產生差額，系統應自動進行尾差調整。
3. 尾差建議調整至本期用電度數最高之房間。
4. 系統應保留尾差調整紀錄，包括調整房號、調整金額與調整原因。

### 需求說明

1. 電費分攤結果應可產生「房客版明細」，方便房東傳給房客。
2. 房客版明細應包含帳單期間、公用電費、戶內電費、該房用電度數、用電比例、公用電費分攤、戶內電費分攤與應繳金額。
3. 電費不應併入房租收入，應獨立列為電費代收或電費回收。
4. 系統應能查看每一期臺電帳單是否已向房客收齊。
5. 若房東已先代繳臺電，系統應能顯示「臺電已繳」與「房客尚未繳清」兩種狀態。

## 4.9 支出管理

### 功能文字說明

支出管理用於記錄所有由房東負擔或暫時墊付的費用，包括水費、垃圾清潔費、維修費、設備汰換、公共耗材、清潔費與其他支出。

由於水費與垃圾清潔費已含在房租內，因此這兩項應列為房東營運支出，不應再向房客另外收取。

### 功能細項設定

支出紀錄應至少包含下列欄位：

| 欄位 | 說明 |
|---|---|
| 支出編號 | 系統自動產生 |
| 支出日期 | 實際付款日 |
| 歸屬月份 | 該筆支出歸屬月份 |
| 支出類型 | 水費、垃圾清潔費、維修、設備、清潔、耗材、稅費、其他 |
| 支出歸屬 | 公共、301、302、303、304 |
| 金額 | 支出金額 |
| 負擔者 | 房東、房客、押金扣抵、共同分攤 |
| 是否已付款 | 是／否 |
| 是否由維修轉入 | 是／否 |
| 關聯維修案件 | 若由維修轉入，需綁定案件 |
| 附件 | 發票、收據、照片 |
| 備註 | 補充說明 |

### 需求說明

1. 支出應可分為公共支出與房間支出。
2. 公共支出可用於計算整體出租淨利，也可在報表中平均分攤至四間房。
3. 房間支出應歸屬到特定房號，用於計算房間別損益。
4. 維修案件完成後，若產生房東負擔費用，應可一鍵轉入支出紀錄。
5. 若維修費由房客負擔，系統應可轉為應收款或押金扣抵，不應直接列為房東支出。

## 4.10 維修管理

### 功能文字說明

維修管理用於記錄房客回報或房東發現之設備損壞、維修處理過程、照片、費用與責任歸屬。此功能主要目的在於保留完整證據，避免日後產生維修費、押金扣抵或退租爭議。

### 功能細項設定

每一筆維修案件應至少包含下列欄位：

| 欄位 | 說明 |
|---|---|
| 維修案件編號 | 系統自動產生 |
| 房號 | 301、302、303、304、公共區域 |
| 回報人 | 租客或房東 |
| 申請日期 | 問題回報日期 |
| 維修類型 | 冷氣、熱水器、門鎖、水管、電器、家具、其他 |
| 問題描述 | 損壞情形文字說明 |
| 損壞照片 | 可上傳多張照片 |
| 處理狀態 | 待處理、已派工、處理中、已完成、取消 |
| 處理日期 | 實際完成日期 |
| 維修廠商 | 廠商名稱與聯絡方式 |
| 維修金額 | 實際支出或報價 |
| 負擔者 | 房東、房客、押金扣抵、共同分攤 |
| 是否轉成支出 | 是／否 |
| 是否轉成應收 | 是／否 |
| 完工照片 | 完修後照片 |
| 收據附件 | 維修收據或發票 |
| 備註 | 補充說明 |

### 需求說明

1. 維修案件應可上傳損壞前照片與完工後照片。
2. 維修案件應可依狀態篩選，例如只看待處理案件。
3. 維修完成後，若由房東負擔，應可一鍵轉入支出紀錄。
4. 若由房客負擔，應可轉成房客應收款或押金扣抵紀錄。
5. 維修紀錄應能從房間頁、租客頁與附件中心查到。
6. 系統應保留維修責任歸屬，避免日後退租爭議。

## 4.11 押金管理

押金管理用於記錄押金收取、保管、扣抵與退還。押金不應視為一般收入，因為押金原則上是房東暫時保管之款項，只有在扣抵欠租、電費、維修費或清潔費時，才轉為實際抵扣項目。

押金紀錄應至少包含下列欄位：

| 欄位 | 說明 |
|---|---|
| 押金紀錄編號 | 系統自動產生 |
| 房號 | 對應房間 |
| 租客 | 對應租客 |
| 租約編號 | 對應租約 |
| 押金應收金額 | 合約約定押金 |
| 押金實收金額 | 實際收到金額 |
| 收取日期 | 實際收到押金日期 |
| 付款方式 | 轉帳、現金、其他 |
| 押金狀態 | 保管中、部分扣抵、已退還、全額扣抵 |
| 扣抵項目 | 欠租、電費、維修、清潔、其他 |
| 扣抵金額 | 實際扣抵金額 |
| 退還金額 | 實際退還房客金額 |
| 退還日期 | 實際退還日期 |
| 附件 | 匯款截圖、扣抵明細、退租點交照片 |
| 備註 | 補充說明 |

### 需求說明

1. 押金不應自動列入租金收入。
2. 押金扣抵時，應記錄扣抵原因與附件。
3. 押金退還時，應記錄退還金額、日期與付款方式。
4. 退租結算時，系統應可彙整未繳房租、未繳電費、維修費、清潔費與應退押金。
5. 押金紀錄應與租約綁定。

附件中心用於集中管理所有上傳至雲端的檔案與照片。附件不應只是單純上傳到資料夾，而應綁定到對應的房間、租客、租約、收租、電費、支出或維修紀錄。

| 附件類型 | 合約、帳單、電表、收據、維修照片、匯款截圖、其他 |

4. 電表照片應與電表抄表管理綁定。

# 五、特殊情境需求

## 5.1 空房期間

若房間為空房，但仍產生用電、維修或清潔支出，系統應允許將費用歸屬為房東負擔。

需求如下：

1. 房間狀態可設為空房。
2. 空房期間若有電表用電，負擔者可設為房東。
3. 空房整修或清潔支出可歸屬該房。
4. 報表中應能區分出租期間支出與空房整修支出。

---

## 5.2 中途入住或退租

若租客在臺電帳單期間中途入住或退租，系統應支援分段計算。

需求如下：

1. 退租當日可新增一次電表抄表紀錄。
2. 新租客入住當日可新增一次電表抄表紀錄。
3. 同一期臺電帳單可拆成多段負擔者。
4. 若沒有分段抄表，系統可允許以日期比例估算，但需標記為人工估算。
5. 估算紀錄需保留備註與附件。

---

## 5.3 電表異常或換表

需求如下：

1. 若本期度數小於上期度數，系統應警示。
2. 系統應提供換表紀錄功能。
3. 換表時可記錄舊表結束度數、新表起始度數與換表日期。
4. 手動調整使用度數時，需填寫調整原因。
5. 調整紀錄應保留在該期帳單中。

---

## 5.4 維修費責任歸屬

需求如下：

1. 維修費可設定為房東負擔、房客負擔、押金扣抵或共同分攤。
2. 若房東負擔，維修案件可轉為支出紀錄。
3. 若房客負擔，維修案件可轉為應收款。
4. 若押金扣抵，維修案件應連結押金管理。
5. 維修案件應保留照片與收據。

---

# 六、資料儲存與雲端需求

1. 所有房間、租客、租約、收租、電費、支出、維修、押金資料應儲存在雲端資料庫。

4. 系統應支援依房號、月份、類型查詢資料。
5. 系統應有基本登入保護，不應公開暴露資料。

# 九、核心判斷原則

系統開發時應遵守下列原則：

1. 總表與報表只是結果，真正資料來源應是逐筆紀錄。
2. 房租收入、電費代收、支出與押金應清楚分帳。
3. 電費應採臺電帳單分攤模式，不採固定單價模式。
4. 水費與垃圾清潔費由房東支應，列入房東支出。
5. 維修紀錄應可串接照片、責任歸屬與支出紀錄。
6. 押金不列為租金收入，只有扣抵時才轉為實際抵扣項目。
7. 每一筆收入、支出、電費、維修與附件都應能追溯到房號與月份。
8. 已結算資料應避免任意修改，若修改需保留紀錄。
9. 第一版應追求簡潔穩定，不做過度複雜的平台功能。
10. 系統應讓房東能快速查詢、回顧與匯出資料。

# 十、最終需求摘要

本網頁 App 的最終目標是建立一套適合小型房東使用的雲端管理系統。系統應能管理 301、302、303、304 四間套房，並完整記錄租客、租約、房租、押金、臺電帳單、電表度數、電費分攤、支出、維修、附件與報表。

系統最重要的客製化重點是電費計算。電費應依臺電帳單進行分攤，先輸入公用電費與戶內電費，再根據各房電表使用度數計算用電比例，最後得出各房應繳電費。公用電費雖然來自整棟公寓，但仍應依 28 號 3 樓內部各房用電比例分攤。

水費與垃圾清潔費已含在房租內，應列為房東營運支出，不應再向房客另外收取。維修費則依責任歸屬決定由房東負擔、房客負擔或自押金扣抵。所有相關照片與收據都應能上傳雲端，並綁定對應紀錄。

第一版系統應優先完成房間、租客、租約、收租、臺電帳單、電表抄表、電費分攤、支出、維修、附件與報表功能。其他如房客登入、自動通知、線上付款與電子簽約可留待第二版再開發。
